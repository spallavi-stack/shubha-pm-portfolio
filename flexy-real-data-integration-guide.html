Flexy — Real-Time Pricing Integration Guide (For the Engineering Team)

This doc exists because the prototype's Price & Cost screen was wired to genuine ComEd pricing data during this project. This is the exact method used, why it worked, and what has to change to serve real pricing to thousands of real customers instead of one demo screen. It's written as a handoff for an engineer picking this up next.

How the prototype fetches real data today

How I (Claude, building this) pulled real prices during this project: a plain server-side HTTPS GET request to ComEd's own API, no browser involved. That's the whole trick, and it's the one fact this entire doc is built around.

ComEd's Hourly Pricing API is public and needs no API key. Two endpoints were used:

  • https://hourlypricing.comed.com/api?type=5minutefeed — the documented endpoint. Supports a custom range via &datestart=YYYYMMDDhhmm&dateend=YYYYMMDDhhmm, returned in JSON, plain text, or RSS. This is what pulled a genuine 5-day window (Jul 13–17, 2026) to build the prototype's "Week" average.
  • https://hourlypricing.comed.com/api?type=day&date=YYYYMMDD — an undocumented but working endpoint that returns a clean 24-value array for a single calendar day. This is what pulled the "Today" fallback and the sampled days (Jan/Mar/May/Jun/Jul 2026) used to build "Month" and "Year."

Full documented options are at https://hourlypricing.comed.com/hp-api/; API terms are at https://hourlypricing.comed.com/comed-hourly-pricing-apis-terms-of-service/ and should be reviewed before any production commitment, since this project only tested it at demo scale.

Why the prototype needs a CORS proxy but I didn't. CORS (Cross-Origin Resource Sharing) is a restriction browsers enforce on their own. When a browser's JavaScript calls fetch() against a domain that doesn't return an Access-Control-Allow-Origin header, the browser blocks the response, ComEd's API doesn't send that header. A server-side request, what fetched every real number in this project, is exempt: it's a browser-only rule. The shipped prototype is a static HTML file with no backend, so it has to fake having one by routing through a public CORS proxy (allorigins.win). That proxy is genuinely unreliable: it returned empty responses when tested directly during this project. This is the first thing to fix in production: put a real backend between the app and ComEd. The CORS problem was always the client's to begin with, and it disappears the moment a server owns the request instead.

The one fact that makes this trivial to scale

ComEd's Real-Time Pricing feed is a single, system-wide price series. Every customer enrolled in Hourly Pricing sees the exact same price at the exact same time, there is no per-customer pricing call. This means serving 10 customers and serving 100,000 customers costs the backend the same one API call. The scaling problem here is not "fetch more," it's "fetch once, store it, and serve it to everyone from your own database." Don't design a per-request-to-ComEd architecture; design a single ingestion pipeline with many readers.

This is different from the two other data sources Flexy depends on:

  • Usage data (Green Button Connect) genuinely is per-customer and requires individual OAuth-style authorization per household. See flexy-technical-feasibility.md for that integration's access model and risk profile; it isn't duplicated here.
  • Vehicle data (Smartcar) is also per-customer, per-vehicle, OAuth-based. Also covered in the feasibility doc.

Only the price feed is shared. That's the one worth optimizing hard for reuse.

Recommended production architecture

ComEd Hourly Pricing API
        │  (server-side GET, every 5-15 min)
        ▼
 Ingestion job (cron / cloud scheduler)
        │
        ▼
 Price time-series store (owned by Flexy)
        │
        ├─► Hourly/weekly/monthly/yearly rollup tables (precomputed)
        │
        ▼
 Flexy's own internal API
        │
        ▼
   Flexy mobile app  (never talks to ComEd directly)

Ingestion job. A small scheduled job (cron, AWS EventBridge, Cloud Scheduler, whatever's already in the stack) hits type=5minutefeed every 5-15 minutes and appends new ticks to a time-series table. A separate one-time backfill job walks type=day&date= across whatever historical range is needed to seed Week/Month/Year views immediately instead of waiting weeks for organic data to accumulate.

Storage. A simple schema is enough to start:

comed_price_ticks (
  ts_utc        TIMESTAMPTZ PRIMARY KEY,
  price_cents   NUMERIC(6,2),
  source        TEXT  -- '5minutefeed' | 'day-backfill'
)

comed_hourly_avg (
  date          DATE,
  hour          SMALLINT,
  avg_price_c   NUMERIC(6,2),
  PRIMARY KEY (date, hour)
)

comed_hourly_avg is a materialized rollup refreshed after each ingestion run, that's what actually powers the app's Today/Week/Month/Year views, so no request ever re-aggregates raw 5-minute data on the fly.

Serving layer. Flexy's own backend exposes its own internal endpoint (e.g. /v1/pricing/today, /v1/pricing/range?window=week) backed by that store, with a short in-memory or Redis cache on the hot "current price" read. The app talks only to this endpoint. This is also what finally makes "live" mean something real: today's prototype badge honestly says "live" or "fallback" because a static file genuinely can't guarantee freshness; a real backend with its own ingestion job can guarantee it and drop that caveat entirely.

Reliability, because ComEd's feed isn't always up

During this project's own research, ComEd's live-prices page displayed this exact banner: "PJM, the regional transmission organization, is experiencing technical difficulties with the 5-minute and hourly price feeds." That's a real, observed failure mode. Production ingestion needs to handle it explicitly:

  • Retry with backoff on a failed fetch; don't let one missed poll silently create a data gap.
  • If a gap persists, serve the last known-good price with an explicit "stale" flag rather than blocking or guessing.
  • Alert on ingestion failures past some threshold (e.g. no new tick in 30 minutes during market hours).
  • Never let the app silently show fabricated or interpolated numbers as if they were real. The badge/fallback pattern already built into the prototype (live vs. fallback vs. modeled, and a "verify this yourself" link to ComEd's own feed) is a UX pattern worth carrying into production as-is.

Migration checklist, prototype → production

  1. Stand up a minimal backend service (even a single scheduled function is enough to start) that owns the ComEd fetch.
  2. Point the ingestion job at type=5minutefeed on a schedule; backfill history with type=day&date=.
  3. Stand up comed_price_ticks and comed_hourly_avg tables (or equivalent).
  4. Replace the app's CORS-proxy fetch with a call to Flexy's own /v1/pricing/* endpoint.
  5. Keep the live/fallback/stale badge and "verify against ComEd" pattern; it's honest UX that holds up in production too.
  6. Layer in Green Button Connect and Smartcar separately once the shared price pipeline is solid, they're per-customer problems and shouldn't block the one piece that scales for free.
  7. Review ComEd's API Terms of Service against actual expected request volume before committing to production traffic.
